# 战报长图合成规范

## 1. 概述

战报长图是一张 **1080px 宽竖版海报**，用于数据战报 / 活动战报的一图式传播。系统从 KV 头图自动取色，与本地字体叠字合成，无需手动配色。

### 合成流水线（7 步）

| 步骤 | 说明 |
|------|------|
| **1. 扫描** | 验证素材目录，确认 KV.jpg 及分区子文件夹存在 |
| **2. 取色** | 对 KV 顶部 55% 做 K-means 聚类，提取 12 色主题 Token |
| **3. 资产准备** | 可选 AI 生成小 Banner 底图（1080×280，无字无人物） |
| **4. Hero 头部** | KV 上叠主标题+副标题+平台 logo，叠加数据区（5 种布局） |
| **5. 分区排版** | 栏头 + 截图（核心资源矩阵/联动活动/玩家好评） |
| **6. 拼接** | 所有 block 竖排无缝拼接 |
| **7. 输出** | JPEG quality=92，`战报_{标题}_{时间戳}.jpg` |

## 3. 取色机制

由 `color_extract.py` 实现，**纯 numpy 无 sklearn 依赖**：

| 项目 | 值 |
|------|-----|
| 采样区域 | KV 顶部 55%（hero 区域） |
| 亮色聚类 | K=8，最小饱和度 0.38 |
| 暗色聚类 | K=4，明度 ≤0.28 |
| 暖色优先 | 橙/珊瑚/玫红色相加权（1.5×），避免只落大面积绿/蓝 |

输出 12 色 Token JSON（`scripts/assets/battle-report/themes/{id}.json`）：

| Token | 用途 |
|-------|------|
| `bg_page` | 页面底色（取自 KV 暗部） |
| `bg_card` | 卡片底色 |
| `bg_card_dark` | 深色卡片底 |
| `accent_primary` | 主强调色（面积最大亮色簇） |
| `accent_secondary` | 辅强调色（与主色相相差 ≥28°） |
| `accent_bright` | 艳丽色块主色（暖橙/冷青优先） |
| `accent_bright_alt` | 艳丽色块辅色（与主色冷暖互补） |
| `text_primary` | 主文字色（与 bg_page 高对比度） |
| `text_secondary` | 辅文字色 |
| `text_on_light` | 浅底文字 |
| `stroke_decor` | 描边装饰色（72% 透明度） |

## 4. 字体配置

战报全部叠字使用本地字体，禁止系统默认回退：

| 角色 | 文件名 | 字重 | 用途 | 字号 |
|------|--------|------|------|------|
| `display_bold` | `display-bold.otf` | Bold/Heavy | 主标题、栏目标题 | 100px |
| `display_medium` | `display-medium.otf` | Medium | 副标题、首发条 | 60px |
| `body_regular` | `body-regular.otf` | Regular | 正文、评论 | — |
| `data_bold` | `data-bold.otf` | Bold | 数据数字（曝光/下载） | 64px |

### 字体扫描逻辑

1. 环境变量 `BATTLE_REPORT_FONT_DISPLAY_BOLD` 等 → 指定的绝对/相对路径
2. `scripts/assets/battle-report/fonts/` → 按角色优先文件名查找
3. 该目录下任意 `.otf/.ttf` → 兜底
4. 均未找到 → 报错退出

四角色可共用同一套字体（如造字工房启黑体的 4 个字重变体）。

## 5. Hero 头部布局

### 文案定位

- 主标题 + 副标题 + logo 落在 KV **下半区**（`y ≥ KV高 × 0.66 + 56px`），避开中上人物
- 首发条 + 数据指标在文案下方，**整体向下延伸**，不顶移标题

### 数据区 5 种布局

| 布局 | 说明 | 特点 |
|------|------|------|
| **plain**（默认） | KV 上直接分色叠字，无底色块 | 简洁，最大依赖 KV 本身 |
| **ai_stage** | 程序化舞台背景 + pill 首发条 + 白字数据 | 暗色渐变 + 底部聚光 + KV rim 光 |
| **framed** | 羊皮纸外框 + 圆角内卡 + 黄标 pill + 描边数值 | 最精致，适合有面板标题的场景 |
| **monument** | 巨幕数字舞台，暗场底光 | 适合有大数字冲击力的战报 |
| **classic** | 斜切底卡 + 居中首发条 + 常规数据行 | 稳妥通用 |

## 6. 分区排版规则

### 6.1 栏头区（Section Banner）

- 尺寸：1080×280（AI 底图）或程序化绘制
- 角色图：464×376，顶对齐（可溢出顶 60px）
- 标题字：88px local font，二次斜切色块装饰

### 6.2 核心资源矩阵（七行固定模板）

```
┌───────┬───────┐
│  01   │  02   │  ← pair（双列并排，同缝满宽）
├───────┴───────┤
│      03       │  ← full（满宽）
├───────┬───┬───┤
│  05   │06 │07 │  ← triple（三列，与 01+02 同缝）
├───────┴───┴───┤
│      04       │  ← full
├───────────────┤
│  其余截图 ...  │  ← remainder（browser→photo→uncategorized→long）
├───────────────┤
│      08       │  ← full（收尾长图）
└───────────────┘
```

### 6.3 联动活动区

- 邮件(左)+活动(右)：等高并排，等比缩放至满宽，不裁切
- 其余截图：flow 自适应排版

### 6.4 玩家好评区

- 默认 grid_2x2 四格矩阵
- 其余截图 flow 自适应

### 6.5 截图分类器

| 类型 | 条件 | 排版 |
|------|------|------|
| 竖图 pair | 宽 600–900，高宽比 ≥1.12 | 两两并排 |
| 长图 | 宽 ≥1800 | 满宽单排 |
| 商店/LZ | 编号奇数→商店，偶数→LZ；否则四角亮度判断 | 分类后排 |

## 7. AI 底图生成（可选）

### 控制开关

| 环境变量 | 默认 | 效果 |
|----------|------|------|
| `BATTLE_REPORT_NANO_BANANA` | `1`（有 key 时） | 总开关 |
| `BATTLE_REPORT_SECTION_BANNER_IMAGE` | `0` | 小 Banner 生图 |
| `BATTLE_REPORT_HERO_DATA_IMAGE` | `0` | 数据区生图 |
| `BATTLE_REPORT_NANO_BANANA_REFRESH` | `0` | 强制重生成 |

### 后端优先级

MICU API（`MICU_API_KEY`）→ baoyu-image-gen → nano-banana → Gemini 回退

## 8. 环境变量速查

| 变量 | 说明 |
|------|------|
| `MICU_API_KEY` | MICU 生图/Vision API 密钥 |

| `BATTLE_REPORT_AI_SCREENSHOT_ORDER` | AI 截图排序（默认 `0`） |
| `BATTLE_REPORT_AI_DESIGN` | AI 数据区排版设计（默认 `0`） |
